# T-Credit 기능정의서

## 1. 개요

T-Credit은 계명대학교 구성원의 비교과적 공헌 활동을 입력, 추천, 승인, 심의하고 누적 Credit을 조회하기 위한 시범운영 웹 시스템이다.

- 대상: 대학혁신지원사업 중심의 1차 시범운영
- 사용자: 공헌자, 승인자, 위원회, 관리자
- 기술 기반: NextJS App Router, TypeScript, Prisma, PostgreSQL, Tailwind CSS
- 데이터 저장: PostgreSQL 실 DB 기준
- 인증/조직 연동: MVP에서는 DB 사용자 계정 전환 방식으로 테스트하며, 향후 학교 SSO/인사 DB 연동으로 교체
- 제도 목적: 구성원의 통상 업무 외 공헌을 즉시 기록하고, 동료 추천과 승인권자 확인을 통해 투명하고 설명 가능한 인정 근거를 축적한다.
- 운영 원칙: 즉각성, 투명성, 공정한 근거, 조직 협력 유도, 승인권자 책임성을 핵심 원칙으로 둔다.
- 인정 범위: 기존 업적평가에 이미 정량 반영된 활동이나 별도 보수가 이미 책정된 활동은 원칙적으로 제외한다.
- MVP 범위: T-Credit 적립, 승인, 조회, 심의 자료 제공까지 포함하며 실제 성과급 산정/급여 연동은 후속 범위로 둔다.

## 2. 사용자 역할

| 역할 | 설명 | 주요 권한 |
| --- | --- | --- |
| CONTRIBUTOR | 공헌 입력자 | 공헌 입력, 본인 공헌 조회, 이의신청 상태 확인 |
| APPROVER | 승인권자 | 배정 공헌 검토, 점수/Tier 조정, 승인/반려 |
| COMMITTEE | 위원회 | 이상 사례, 이의신청, 비공개 의견, 표본감사 검토 |
| ADMIN | 관리자 | 사용자, 조직, 승인권자, 예산, 임계값 관리 |

관리자는 모든 역할 메뉴에 접근할 수 있다.

추천인은 사용자 역할이 아니라 공헌 입력 시 공헌 건별로 지정되는 관계이다.

## 2.1 제도 참여 관계

- 공헌자는 본인의 추가 공헌 활동을 직접 입력한다.
- 관련 부서 구성원은 공헌 내용을 확인하고 추천 의견 또는 위원회용 비공개 의견을 남길 수 있다.
- 승인자는 해당 공헌을 알고 설명할 수 있는 범위에서 승인/반려를 처리한다.
- 위원회는 모든 건을 전수 확인하지 않고 이상 사례, 이의신청, 비공개 의견, 무작위 표본감사를 중심으로 심의한다.
- 본부/관리자는 전체 발행 Credit, 예산 풀, 승인권자별 발행 행태를 관리한다.

## 3. 공통 기능

### 3.1 레이아웃

- 좌측 사이드바에 사용자 역할에 맞는 메뉴만 표시한다.
- 좌측 사이드바 하단에 `My Role` 영역을 두고 현재 사용자의 역할을 표시한다.
- 우측 상단에는 현재 사용자 정보와 누적 Credit을 표시한다.
- 누적 Credit은 현재 사용자 기준 승인 완료 공헌의 `Approval.finalCredit` 합계로 계산한다.
- 상단 사용자 이니셜 원형에는 현재 사용자가 처리해야 할 건수를 숫자 배지로 표시한다.
- 상단 배지 숫자는 `APPROVER` 역할이 있으면 승인 대기 공헌 수를 우선 표시하고, `CONTRIBUTOR`만 있으면 추천 의견 입력 대기 수를 표시한다.
- `ADMIN + APPROVER` 사용자의 상단 배지는 전체 승인 대기 공헌 수를 표시한다.
- 일반 `APPROVER` 사용자의 상단 배지는 본인 소속 부서의 승인 대기 공헌 수를 표시한다.
- 좌측 사이드바의 `공헌 내역` 메뉴에는 현재 사용자가 추천인으로 지정되어 아직 의견을 입력하지 않은 추천 요청 수를 배지로 표시한다.
- 좌측 사이드바의 `승인 처리` 메뉴에는 현재 승인자가 처리해야 할 승인 대기 공헌 수를 배지로 표시한다.

### 3.2 계정 전환

- 상단 사용자 영역을 클릭하면 DB에 등록된 사용자 목록을 표시한다.
- 사용자 선택 시 쿠키에 선택 사용자 ID를 저장한다.
- 계정 전환 후 현재 경로에 필요한 역할이 없으면 홈 화면으로 이동한다.
- 계정 전환 후 화면과 메뉴는 즉시 새 사용자 기준으로 갱신한다.

## 4. 메뉴 및 화면 기능

### 4.1 운영 현황 `/`

- 전체 공헌 운영 현황을 대시보드로 표시한다.
- 승인 Credit, 승인 완료 건수, 승인 대기 건수, 위원회 큐 건수를 표시한다.
- 예산 풀 정보를 표시한다.
- 최근 공헌 로그 표를 표시한다.
- 최근 공헌 로그는 계정 역할에 따라 표시 범위를 제한한다.
- `CONTRIBUTOR`는 본인이 입력한 공헌 로그만 표시한다.
- `APPROVER`는 본인 소속 부서의 공헌 로그만 표시한다.
- `ADMIN`은 전체 공헌 로그를 표시한다.
- 공헌자명, 관련 부서명은 DB 사용자/조직 정보 기준으로 표시한다.
- 관련 부서 구성원에게 공개된 공헌은 향후 동료 추천과 비공개 의견 입력 대상으로 활용한다.

### 4.2 공헌 입력 `/contributions/new`

- 공헌 제목, 유형, 활동일, 관련 부서, Effort Tier, 설명을 입력한다.
- 공헌 제목은 최대 255자까지 입력할 수 있다.
- 활동일은 현재 날짜 이후의 미래 날짜를 선택할 수 없다.
- 공헌 유형은 다음 6개 유형을 기본값으로 제공한다.
  - 통상업무 외 중요 업무 참여(협력)
  - 기존 업무의 문제 해결(개선)
  - 신규 및 고난이도 업무(도전)
  - 뛰어난 업무 성과 창출(탁월)
  - 대외적 위상 제고(긍지)
  - 기타 공헌 활동(헌신)
- 동료추천인은 선택 입력 항목으로 제공한다.
- 동료추천인은 여러 명을 추가 등록할 수 있다.
- 동료추천인을 지정하는 경우 추천인 성명과 추천인 부서를 입력하거나 DB 사용자 목록에서 선택한다.
- 관련 부서 구성원에게 공헌 내용이 공개되며, 관련 부서 기준 추천/비공개 의견 입력이 가능하도록 한다.
- 공헌 제출 시 지정된 추천인에게 메일 또는 시스템 알림을 발송한다.
- 추천인은 알림을 받은 뒤 해당 공헌에 대한 추천 의견을 입력할 수 있다.
- 추천인은 공개 추천 의견 또는 승인자/위원회 확인용 비공개 의견을 입력할 수 있다.
- 관련 부서 목록은 DB의 `OrgUnit` 기준으로 표시한다.
- 참여, 성과, 효과 점수를 체크하여 예상 Credit을 미리 계산한다.
- Credit 산식은 `(참여 + 성과 + 효과) * Tier 배수`이다.
- Tier 배수는 `L1 x1`, `L2 x3`, `L3 x6`이다.
- 활동일 기준 30일 이내면 `RECOMMENDING`, 30일 초과면 `UNBILLABLE` 상태로 저장한다.
- 제출된 공헌은 DB의 `Contribution` 테이블에 저장한다.
- MVP에서는 사용자가 추천인을 직접 지정하는 방식부터 구현하고, 관련 부서 전체 알림과 추천 가능 범위 자동 산정은 후속 고도화로 둔다.

### 4.3 공헌 내역 `/contributions/mine`

- 현재 사용자의 공헌 목록만 표시한다.
- 누적 승인 Credit, 입력 공헌 건수, 이의신청 건수를 표시한다.
- 본인 공헌별 추천 의견, 승인 상태, 이의신청 상태를 표시한다.
- 추천자명과 이의신청 정보는 DB 기준으로 표시한다.
- 현재 사용자가 추천인으로 지정된 공헌 목록을 별도 영역에 표시한다.
- 추천인으로 지정된 사용자는 해당 공헌에 대해 공개 추천 의견 또는 비공개 의견을 입력할 수 있다.
- 비공개 의견은 공헌자에게 공개하지 않고 승인자/위원회 검토 자료로만 활용한다.
- 추천 의견 제출 전 상태는 `REQUESTED`, 제출 후 상태는 `SUBMITTED`로 관리한다.
- 사이드바 `공헌 내역` 메뉴 배지는 `Recommendation.recommenderId = 현재 사용자 ID`이고 `Recommendation.status = REQUESTED`인 건수를 표시한다.
- 추천 의견 제출 후 해당 `Recommendation.status`가 `SUBMITTED`로 변경되면 `공헌 내역` 메뉴 배지 숫자는 즉시 감소해야 한다.
- AI 요약 영역은 현재 규칙 기반 안내 문구를 표시하며 실제 AI 연동은 후속 범위이다.

### 4.4 승인 처리 `/approvals`

- 승인 대기 공헌 1건을 승인 작업대로 표시한다.
- 공헌 상세, 추천/비공개 의견, 공헌자, 관련 부서를 표시한다.
- 사이드바 `승인 처리` 메뉴 배지는 `Contribution.status = PENDING_APPROVAL`인 공헌 수를 표시한다.
- 일반 `APPROVER`의 승인 처리 배지는 `Contribution.relatedOrgUnitCode = 현재 사용자 소속 조직 코드` 조건을 추가하여 본인 소속 부서 공헌만 계산한다.
- `ADMIN + APPROVER`의 승인 처리 배지는 소속 부서 조건 없이 전체 승인 대기 공헌을 계산한다.
- 승인자는 참여, 성과, 효과 점수와 최종 Tier를 조정할 수 있다.
- 승인자는 해당 공헌을 제3자가 보아도 설명 가능한 추천인지 확인하는 책임을 가진다.
- 승인자가 내용을 알지 못하거나 설명할 수 없는 공헌은 승인 대상이 아니며 반려 또는 보류 처리한다.
- 증빙 첨부는 MVP 필수 항목으로 두지 않고, 승인자 책임과 추천/의견 로그를 근거로 삼는다.
- 승인 시 최종 Credit을 계산하고 `Approval`에 저장한다.
- 반려 시 최종 Credit은 0으로 저장한다.
- 승인/반려 의견은 필수로 입력한다.
- 승인/반려 처리 후 `Contribution.status`를 함께 갱신한다.
- 승인/반려/점수 조정 내역은 `AuditEvent`에 기록한다.

### 4.5 승인자 조회 `/approver-insights`

- 승인율, 발행 Credit, 반려 건수를 표시한다.
- 월별 승인 처리 현황 예시 차트를 표시한다.
- 자동 경고 예시를 표시한다.
- 승인 내역 표를 DB 기준으로 표시한다.
- 공헌자명은 DB 사용자 정보 기준으로 표시한다.
- 특정 시기 집중 처리, 특정 공헌자 집중, 특정 부서 집중, 유형별 편중을 식별할 수 있는 지표를 제공한다.
- 평균 대비 지나치게 높거나 낮은 승인율과 발행 Credit 편차를 위원회 점검 자료로 표시한다.

### 4.6 위원회 심의 `/committee`

- 심의 대기 건수, 비공개 의견 건수, 표본감사 비율을 표시한다.
- 이상 사례 큐를 `CommitteeReview` 기준으로 표시한다.
- 무작위 표본감사 대상 공헌을 표시한다.
- 이의신청 목록은 DB의 `Appeal` 기준으로 표시한다.
- 공헌자명은 DB 사용자 정보 기준으로 표시한다.
- 이상 사례 기준은 다음 항목을 포함한다.
  - 승인권자가 한 사람에게 전체 권한의 일정 비율 이상을 배분한 경우
  - 전체 입력 건 중 반려 비율이 일정 수준 이상인 공헌자의 신청 내용
  - 진행 과정에서 비공개 의견 또는 이의신청이 접수된 경우
  - 동일하거나 유사한 내용으로 반복 신청 또는 부서별 중복 승인 요청이 발생한 경우
  - 승인율이 평균 대비 지나치게 높거나 낮은 경우
  - 공헌 신청 또는 승인 활동이 특정 기간에 집중되는 경우

### 4.7 관리자 `/admin`

- 사용자 수, 조직 수, 승인권자 수, 예산 정보를 표시한다.
- 사용자 및 권한 목록을 DB의 `User`와 `OrgUnit` 기준으로 표시한다.
- 예산 풀은 DB의 `BudgetPool` 기준으로 표시한다.
- 심의 임계값은 현재 운영 규칙 예시로 표시한다.
- 본부가 정한 연도/학기별 총 예산과 1 Credit 명목가치에 따라 전체 발행 가능 Credit을 산정한다.
- 부서/단과대/팀별 승인권 배분은 인원 비중과 협력가중계수 등 운영 정책을 반영할 수 있도록 확장한다.
- 부서장과 팀장은 배정된 Credit의 일정 비율을 타 부서 협력 공헌에 배분하도록 정책값을 둘 수 있다.

## 5. 데이터 모델

주요 테이블은 Prisma schema 기준으로 정의한다.

| 모델 | 용도 |
| --- | --- |
| `User` | 사용자, 교번/사번, 이메일, 직군, 직책, 소속 조직, 역할 |
| `OrgUnit` | 부서/단과대/팀 조직 계층 |
| `Contribution` | 공헌 입력 내용, 유형, 활동일, 관련 부서, 상태, 예상 Credit |
| `Recommendation` | 공헌별 추천인 지정, 추천 요청 상태, 알림 발송 정보, 공개/비공개 의견 |
| `Approval` | 승인 점수, 최종 Tier, 최종 Credit, 승인/반려 의견 |
| `Appeal` | 공헌자 이의신청 |
| `CommitteeReview` | 위원회 이상 사례 및 검토 메모 |
| `BudgetPool` | 연도/학기별 예산, Credit 명목가치, 발행 한도 |
| `AuditEvent` | 주요 상태 변경 및 승인/반려/조정 로그 |

### 5.1 추천인 데이터 구조 검토

- 한 공헌에는 여러 추천인을 지정할 수 있으므로 `Contribution`과 `Recommendation`은 1:N 관계로 유지한다.
- `Recommendation`은 추천인 사용자 ID를 `recommenderId`로 저장한다.
- 동일 공헌에 동일 추천인이 중복 지정되지 않도록 `contributionId + recommenderId` 기준으로 유일성을 둔다.
- 추천 요청 상태는 `REQUESTED`, `SUBMITTED`, `CANCELED`로 관리한다.
- 알림 채널은 `EMAIL`, `IN_APP`으로 구분한다.
- 알림 발송 시각은 `notifiedAt`, 추천 의견 제출 시각은 `submittedAt`에 저장한다.
- 추천 의견은 요청 시점에는 비어 있을 수 있으므로 선택값으로 저장한다.
- 비공개 의견 여부는 `isPrivate`로 구분한다.

### 5.2 예산 및 승인권 배분 데이터 검토

- `BudgetPool`은 총 예산, 1 Credit 명목가치, 전체 발행 한도를 관리한다.
- 향후 부서별 승인권 배분을 위해 `OrgUnit` 단위의 배정 Credit, 협력가중계수, 타 부서 배분 비율 정책을 추가할 수 있다.
- 승인권자별 연간 발행 Credit, 발행 한도, 할인계수, 실효가치는 승인 남발 억제 정책을 위해 별도 집계하거나 확장 테이블로 관리한다.
- MVP에서는 `BudgetPool`과 승인 내역 집계를 통해 현황을 표시하고, 자동 할인/페널티 산정은 후속 고도화 범위로 둔다.

## 6. 상태 및 산식

### 6.1 공헌 상태

| 상태 | 의미 |
| --- | --- |
| `DRAFT` | 초안 |
| `RECOMMENDING` | 추천 대기 |
| `PENDING_APPROVAL` | 승인 대기 |
| `APPROVED` | 승인 완료 |
| `REJECTED` | 반려 |
| `UNBILLABLE` | 입력기한 초과 |
| `UNDER_REVIEW` | 위원회 검토 |

### 6.2 Credit 산식

```text
최종 Credit = (참여 점수 + 성과 점수 + 효과 점수) * Effort Tier 배수
```

- 참여, 성과, 효과 점수는 각각 0 또는 1이다.
- Effort Tier 배수는 L1=1, L2=3, L3=6이다.
- 반려 시 최종 Credit은 0이다.

### 6.3 공헌 유형별 인정 기준

## 7. 권한 규칙

- 메뉴는 현재 사용자 역할 기준으로 노출한다.
- `ADMIN`은 모든 메뉴 접근 권한을 가진다.
- 공헌 내역은 현재 사용자 본인의 공헌만 표시한다.
- 승인 처리는 승인권자 역할 사용자에게만 메뉴를 노출한다.
- 위원회 심의는 위원회 또는 관리자 역할 사용자에게만 메뉴를 노출한다.
- 추천 의견 작성자는 별도 역할이 아니라 공헌별 추천인 지정 정보로 판단한다.
- 추천인은 `Recommendation`에 등록된 사용자만 해당 공헌의 추천/비공개 의견을 입력할 수 있다.
- 비공개 의견은 승인자와 위원회 정책에 따라 제한적으로 조회한다.
- 관련 부서 구성원에게 공개된 공헌은 향후 부서 단위 추천 가능 범위 정책에 따라 추천 입력 권한을 부여할 수 있다.
- 위원회는 비공개 의견과 이의신청을 확인할 수 있으나, 공헌자에게 비공개 의견의 작성자와 내용을 직접 공개하지 않는다.

## 8. 초기 설정 및 DB 운영

- PostgreSQL은 Docker Compose로 실행한다.
- `.env`의 `DATABASE_URL`을 Prisma 연결 문자열로 사용한다.
- 최초 설정 순서:

```bash
docker compose up -d
npm run db:push
npm run db:seed
npm run dev
```

- `db:push`는 Prisma schema 기준으로 테이블/컬럼을 생성한다.
- `db:seed`는 조직, 사용자, 공헌, 추천, 승인, 이의신청, 위원회, 예산 초기 데이터를 생성한다.

## 9. 테스트 시나리오

### 9.1 단위 테스트

- Credit 산식 검증
- 30일 입력 제한 검증
- 역할별 접근 권한 검증
- 추천 의견 입력 대기 배지 계산 검증
  - `Recommendation.recommenderId`가 현재 사용자 ID이고 `status = REQUESTED`인 건수만 카운트한다.
  - `SUBMITTED`, `CANCELED` 상태 추천 요청은 카운트하지 않는다.
- 승인 대기 배지 계산 검증
  - `Contribution.status = PENDING_APPROVAL`인 건수만 카운트한다.
  - 일반 `APPROVER`는 현재 사용자 소속 조직 코드와 `Contribution.relatedOrgUnitCode`가 일치하는 건수만 카운트한다.
  - `ADMIN + APPROVER`는 조직 조건 없이 전체 `PENDING_APPROVAL` 건수를 카운트한다.
  - `APPROVER` 또는 `ADMIN` 역할이 없는 사용자의 승인 대기 배지는 0으로 계산한다.
- 상단 사용자 배지 우선순위 검증
  - `APPROVER` 역할이 있으면 승인 대기 배지 수를 표시한다.
  - `APPROVER` 역할이 없고 `CONTRIBUTOR` 역할만 있으면 추천 의견 입력 대기 수를 표시한다.

### 9.2 통합 테스트

- 공헌 입력 후 DB 저장 확인
- 공헌 입력 시 추천인을 여러 명 지정하면 추천 요청 데이터가 추천인별로 생성되는지 확인
- 추천인 지정 시 메일 또는 시스템 알림 발송 정보가 기록되는지 확인
- 활동일 30일 초과 시 `UNBILLABLE` 저장 확인
- 승인 처리 후 `Approval`, `Contribution.status`, `AuditEvent` 반영 확인
- 계정 전환 후 메뉴와 누적 Credit 변경 확인
- 계정 전환 후 상단 사용자 배지, `공헌 내역` 메뉴 배지, `승인 처리` 메뉴 배지가 새 사용자 기준으로 다시 계산되는지 확인
- 추천 의견 제출 후 `Recommendation.status`가 `SUBMITTED`로 바뀌고 상단/사이드바 추천 대기 배지가 감소하는지 확인
- 승인 처리 후 `Contribution.status`가 `APPROVED` 또는 `REJECTED`로 바뀌고 상단/사이드바 승인 대기 배지가 감소하는지 확인

### 9.3 화면 테스트

- 공헌 입력 화면 필수값 검증
- 승인 처리 화면 점수/Tier 변경에 따른 Credit 미리보기 검증
- 공헌 내역에서 본인 공헌만 표시되는지 확인
- 공헌 내역에서 내가 추천인으로 지정된 공헌과 의견 입력 상태가 표시되는지 확인
- 추천인이 비공개 의견을 제출하면 `Recommendation.isPrivate`, `comment`, `status`, `submittedAt`이 반영되는지 확인
- `CONTRIBUTOR` 계정에서 상단 사용자 이니셜 배지와 `공헌 내역` 메뉴 배지가 추천 의견 입력 대기 건수와 일치하는지 확인
- 일반 `APPROVER` 계정에서 상단 사용자 이니셜 배지와 `승인 처리` 메뉴 배지가 본인 소속 부서 승인 대기 건수와 일치하는지 확인
- `ADMIN + APPROVER` 계정에서 상단 사용자 이니셜 배지와 `승인 처리` 메뉴 배지가 전체 승인 대기 건수와 일치하는지 확인
- 배지 숫자가 100건 이상이면 `99+` 형식으로 표시되는지 확인
- 위원회 화면에서 이상 사례와 이의신청이 DB 기준으로 표시되는지 확인
- 관리자 화면에서 사용자/조직/예산이 DB 기준으로 표시되는지 확인

## 11. PPTX 대조 결과

### 11.1 기능정의서에 반영한 누락 항목

- 제도 도입 목적과 운영 원칙(즉각성, 투명성, 공정한 근거)을 개요에 추가했다.
- 공헌 유형 6종과 유형별 인정/제외 기준을 추가했다.
- 관련 부서 공개, 관련 부서 구성원 추천/비공개 의견, 승인권자 책임 원칙을 추가했다.
- 승인자별 조회와 위원회 심의의 집중도/편중/기간 집중/중복 신청 등 이상징후 기준을 구체화했다.
- 예산 풀, 1 Credit 명목가치, 부서별 승인권 배분, 타 부서 협력 배분 정책을 추가했다.
- 추천 남발과 승인 남발 억제를 위한 운영 정책과 후속 고도화 항목을 추가했다.
- PPTX의 추진일정을 단계적 구축 일정으로 추가했다.

### 11.2 PPTX에는 없지만 MVP에 필요하여 유지한 항목

- DB 사용자 계정 전환은 실제 운영 기능이 아니라 SSO 연동 전 테스트용 기능으로 유지한다.
- Docker Compose, Prisma `db:push`, `db:seed` 절차는 개발/시범운영 초기 설정 항목으로 유지한다.
- AI 요약은 PPTX에 제시되어 있으나 MVP에서는 실제 AI 연동 전 규칙 기반 안내 또는 준비 중 상태로 유지한다.
- 실제 성과급 지급, 급여 시스템 연동, 자동 할인계수 적용은 PPTX의 제도 방향에는 포함되지만 MVP 구현 범위에서는 후속 고도화로 분리한다.
